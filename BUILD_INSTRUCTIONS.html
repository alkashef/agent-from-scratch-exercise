Hands-on Exercise

Build an AI Agent from Scratch

You will implement the core of a real MCP (Model Context Protocol) agent — the same pattern used by Claude Desktop and other production AI assistants. By the end, you will have a working to-do chatbot driven by a local LLM.

1. Context — What Is an Agent?

An AI agent is a program that connects a language model (LLM) to tools in the real world. The LLM reads the user's message and decides which tool to call and with what arguments. The agent then runs that tool and returns the result.

In this exercise the agent has one tool: add_task. It saves a to-do item to a CSV file. The user types a natural-language message; the agent figures out what the task is and stores it — no manual form-filling required.

The four-step agent loop

Every MCP agent follows the same loop, regardless of which tools it has:

1

Host fetches tools

The host connects to the MCP server and asks "what tools do you have?". The server replies with the name, description, and input schema of every tool. No tools are hardcoded anywhere in the host.

2

Host presents tools to the LLM

The host sends the user's message and the tool list to the LLM via the function-calling API. The LLM can now see both what the user wants and what actions are available.

3

LLM decides which tool to call

Instead of replying with plain text, the LLM responds with a structured tool call: the name of the tool and the arguments it wants to pass. The LLM never runs the tool itself — it only decides.

4

Host executes the tool via MCP

The host validates the LLM's arguments, then forwards the tool call to the MCP server using the MCP protocol. The server runs the tool and returns a result. The host shows the result to the user.

2. What To Do

Before you write a single line of code

1

Create two separate git repositories — one for todo-mcp-server/ and one for todo-mcp-host/.

Each component is an independent deployable unit and should have its own version history.

2

Read both README files in full — todo-mcp-server/README.md and todo-mcp-host/README.md.

They cover prerequisites, environment setup, how to install dependencies, how to run each component, and what the expected behaviour looks like. Do not skip this step.

3

Set up the MCP server in Claude Desktop and run it first.

Before writing any host code, get the server working end-to-end with a real MCP client so you can see the protocol in action:

  1. Follow the Setup section in todo-mcp-server/README.md to install dependencies and start the server: python -m csv_todo_mcp.server
  2. Connect the running server to Claude Desktop using its interface.
  3. Open a new conversation in Claude Desktop. Ask: "What tools do you have?" — confirm add_task appears.
  4. Ask Claude to add a to-do item, e.g. "Add a task to call Alice by Friday." Open todo-mcp-server/data/todo.csv and confirm the row was written.

This step has no code to write — it is purely observational. Once you have seen the tool called and the CSV updated, you understand exactly what the host you are about to build must do. Then come back here and continue to step 4.

4

Read every source file in todo-mcp-host/ top-to-bottom before touching anything.

Every class and method has a docstring that explains its contract. The files you will not implement (config.py, chat_ui.py, logging_config.py, main(), App.run()) are the scaffolding that holds everything together — you need to understand them to implement the parts that are missing.

Implement these files — in order

Each file below has one or more sections marked:

**** START HERE ****   …your code…   **** END HERE ****

Tip: Inside each marked section you will find detailed hints and a worked example showing the exact API calls. Read the hints before writing anything.
1
todo-mcp-host/prompts/system_prompt.md
The instructions you give the LLM before it sees any user message. This is what tells the model to call add_task and how to fill the fields.
Write: A system prompt that tells the LLM to act as a to-do assistant, call the add_task tool for actionable messages, fill the correct fields (description, status, category, people, deadline), and reply with plain text when no task is present. Include the {TODAY} placeholder for the current date.
2
todo-mcp-host/src/mcp_todo_host/task_parser.py
Validates the arguments the LLM produced and applies defaults for missing optional fields. This sits between the LLM output and the MCP call — always sanitise before you act.
Implement: TaskParser.validate() — check that description is non-empty, apply defaults for the four optional fields, and return a clean 5-key dict.
3
todo-mcp-host/src/mcp_todo_host/llm_client.py
Wraps the local Ollama model. Builds the message list (system + user), passes the tool definitions, and extracts the tool call from the response. This is where the LLM's decision is captured.
Implement: LlmClient.generate() — inject today's date into the prompt, build the messages list, call self._client.chat(..., tools=tools), and return {"name": ..., "arguments": ...}.
4
todo-mcp-host/src/mcp_todo_host/mcp_client.py
The MCP protocol client. Connects to the MCP server over HTTP, retrieves the tool list, and calls tools — all using the official MCP Python SDK. This file has five sections to implement.
Implement (5 sections):
• _mcp_tool_to_ollama(tool) — convert an MCP Tool object to Ollama format.
• list_tools() — synchronous wrapper using asyncio.run().
• _list_tools_async() — call session.list_tools() and convert results.
• call_tool() — synchronous wrapper using asyncio.run().
• _call_tool_async() — call the tool and parse the JSON result.

Note: _with_session() is already implemented — use it inside the two async methods.
5
todo-mcp-host/main.py
The orchestrator. App.process() is the agent loop itself — the method that connects all four steps together in one place.
Implement: App.process() — call list_tools() → generate() → validate() → call_tool() → return the success string. This is the payoff: all the pieces you built come together here.
Do not edit any file inside todo-mcp-server/.

3. How to Check Your Work

Follow the startup instructions in todo-mcp-host/README.md to run both components. Once running, use the scenarios below to verify that the agent behaves correctly.

# What to check What success looks like
1 Start both components Streamlit chat UI opens in the browser with no Python errors in the terminal
2 Type a task: "Add a task to call Alice about the project by Friday" Agent replies: "Task added: call Alice about the project (saved at ./data/todo.csv)"
3 Open todo-mcp-server/data/todo.csv File contains a new row with description, status=todo, people=Alice, deadline=Friday
4 Type a greeting: "Hello!" Agent replies with plain text — no tool called, no CSV row added
5 Stop the MCP server and send a task Agent shows "Error: MCP server unreachable" instead of crashing
Tip: Check todo-mcp-host/logs/host.log when something goes wrong. Every step of the agent loop is logged — you will see exactly where the failure is.
